# RAG Service: Продвинутая вопросно-ответная система

## 1. Обзор проекта

Этот проект представляет собой асинхронный микросервис, реализующий **Retrieval-Augmented Generation (RAG)**. Он предназначен для ответов на вопросы по внутренней базе знаний, готов к промышленной эксплуатации и развертыванию в Docker-контейнере.

Сервис использует векторную базу данных (PostgreSQL + `pgvector`) для эффективного поиска релевантной информации и API большой языковой модели (LLM) для синтеза точных и осмысленных ответов.

## 2. Особенности реализации и архитектура

*   **Микросервисная архитектура:** Сервис построен на `FastAPI`, что обеспечивает высокую производительность и асинхронную обработку запросов. Он полностью контейнеризирован с помощью `Dockerfile` для легкого развертывания.
*   **Продвинутый двухэтапный RAG-пайплайн:**
    1.  **Retrieval:** Быстрый поиск широкого набора чанков-кандидатов в `pgvector` с использованием HNSW-индекса.
    2.  **Re-ranking с Chain-of-Thought:** Для переранжирования кандидатов используется LLM (GigaChat) через механизм **Function Calling**. Модель не просто выбирает лучшие чанки, но и предоставляет **обоснование** своего выбора, что делает процесс прозрачным и повышает качество итогового ответа.
*   **Интеллектуальное кэширование:** Реализована двухуровневая система кэширования на основе векторной близости запросов:
    *   **Cache Hit (точное совпадение):** При очень высокой схожести вопроса с уже имеющимся в кэше, сервис мгновенно возвращает готовый ответ.
    *   **Smart Shortcut (умный ярлык):** При средней схожести сервис не запускает полный RAG-цикл, а берет из кэша ID релевантных чанков и "склеивает" полный контекст, что значительно экономит время и ресурсы.
*   **Автоматическое применение схемы БД:** Скрипт `db_schema.sql` автоматически применяется при старте приложения, создавая необходимые таблицы и индексы.

## 3. Ключевые технологии

*   **Бэкенд:** Python, FastAPI, aiohttp, Uvicorn
*   **Базы данных:** PostgreSQL с расширением `pgvector`
*   **ML/NLP:** RAG, Vector Search (HNSW), LLM-as-a-Judge (для re-ranking), Function Calling, Embeddings
*   **Оркестрация:** Docker, Docker Compose

## 5. Установка и запуск

Рекомендуемый способ запуска — через Docker.

### 5.1. Предварительные требования

*   Установленный Docker и Docker Compose.
*   Запущенный экземпляр PostgreSQL с установленным расширением `pgvector`.

### 5.2. Настройка окружения

1.  Создайте файл `.env` в директории `ai_portfolio/rag_service/` по следующему шаблону:

    ```env
    # .env
    
    # URL для подключения к вашей базе данных PostgreSQL
    DATABASE_URL="postgresql://user:password@hostname:5432/dbname"
    
    # Авторизационный токен для GigaChat API
    GIGACHAT_AUTH_TOKEN="your_gigachat_api_token_here"
    ```

2.  **Важно:** Убедитесь, что данные (векторы и тексты чанков) уже загружены в вашу базу данных. Этот сервис отвечает только за обработку запросов (Query-side) и не включает в себя процесс загрузки данных (Ingestion).

### 5.3. Запуск сервиса

Выполните команду из корневой директории вашего проекта:

```bash
docker-compose up rag-service
```
*(предполагается, что в вашем основном `docker-compose.yml` есть сервис с именем `rag-service`, использующий этот `Dockerfile`)*

Или, если вы хотите запустить сервис отдельно, находясь в папке `ai_portfolio/rag_service/`:
```bash
docker build -t rag-service-image .
docker run -p 8002:8002 --env-file .env rag-service-image
```

Сервис будет доступен по адресу `http://localhost:8002`.

## 6. Использование API

Сервис реализует протокол **JSON-RPC 2.0** и ожидает `POST` запросы на корневой эндпоинт `/`.

### Пример запроса

Для того чтобы задать вопрос, отправьте `POST` запрос со следующим телом:

```json
{
    "jsonrpc": "2.0",
    "id": "1",
    "method": "tools/call",
    "params": {
        "name": "answer_question",
        "arguments": {
            "query": "Как работает интеллектуальное кэширование в этом сервисе?"
        }
    }
}
```

### Пример с `curl`

```bash
curl -X POST http://localhost:8002/ \
-H "Content-Type: application/json" \
-d '{
    "jsonrpc": "2.0",
    "id": "1",
    "method": "tools/call",
    "params": {
        "name": "answer_question",
        "arguments": {
            "query": "Как работает интеллектуальное кэширование в этом сервисе?"
        }
    }
}'
```

### Пример ответа

```json
{
    "jsonrpc": "2.0",
    "id": "1",
    "result": {
        "content": [
            {
                "type": "text",
                "text": "\"Интеллектуальное кэширование работает в двух режимах: при очень высокой схожести запроса с кэшированным (Cache Hit) возвращается готовый ответ, а при средней схожести (Smart Shortcut) используются ID релевантных чанков из кэша для формирования полного контекста без запуска всего RAG-пайплайна.\""
            }
        ],
        "isError": false
    }
}
```
